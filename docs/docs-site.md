# The documentation site — tutorial, cookbook, generated reference

Status: design accepted 2026-07-04 (the documentation-site leg of roadmap Phase 35;
Maven Central, the Gradle plugin, and official images are that phase's other legs and
remain open). Two slices: the generator + the authored content, then the generated
reference + publishing.

TesseraQL's documentation already exists and is good: 38 cookbook and design documents
under `docs/`, a deep JSON Schema of the whole YAML surface, and an error taxonomy with
~180 `TQL-*` codes living next to the code that raises them. What is missing is the
front door: one place to read it, navigate it, search it — and reference pages that are
**generated from those machine-readable sources instead of hand-maintained**.

## Decisions

- **A static site, generated by a build-only Maven module** (`tesseraql-docs-site`).
  No runtime, no server, no Node/Python toolchain: the generator is plain Java in the
  reactor, and the output under `target/site` is plain files. The module is never
  published as an artifact.
- **Markdown via commonmark** (+ GFM tables, heading anchors) — already in the tree at
  a pinned version as the Studio doc portal's renderer (`DocMarkdown`), so this adds no
  new dependency family.
- **The site wears Hypermedia Components.** The CSS ships from the version-pinned
  webjar into the site's own assets: the documentation of the design system is styled
  by the design system, fully self-contained, no CDN.
- **Curated navigation with a completeness guard.** A manifest maps every `docs/*.md`
  into a section (Tutorial / Building applications / Security & identity / Operations
  / Platform / Design documents) or excludes it explicitly; a unit test fails when a
  new document is neither — a doc can never silently miss the site.
- **Cross-links just work**: `*.md` links rewrite to `.html`; every href the site emits
  is relative, so the site works under `https://<org>.github.io/tesseraql/` and on a
  local `file://` open alike.

## The generated reference (slice 2)

- **The YAML surface**, from `schema/tesseraql-v1.schema.json` (the same schema the
  editors use): one page per top-level document kind, each property with its type,
  constraints, allowed values, and description — regenerated on every build, so it
  cannot drift from what the loader accepts (the schema itself is already drift-guarded
  against the linter by `SchemaSyncTest`).
- **The error-code index**, scanned from the main source trees: every `TQL-<DOMAIN>-<n>`
  literal with the module and file that raises it, grouped by domain. The scan is over
  our own code, so it is deterministic; where a cookbook page documents the code, the
  index links to it. Honest about coverage: an undocumented code still appears — that
  is the point of an index.
- A **search box** over a generated JSON index (title, headings, first paragraph) with
  a ~40-line client-side filter. No external search service, no heavyweight JS.

## Publishing

`.github/workflows/docs-site.yml` builds the module and deploys `target/site` to
**GitHub Pages** on every push to `main` (`actions/upload-pages-artifact` +
`actions/deploy-pages`). One operator step, documented in the workflow header: the
repository's Pages source must be set to "GitHub Actions". The site is versionless and
tracks `main` — release-pinned snapshots can ride the Phase 35 release train later if
wanted (out of scope here, noted).

## Out of scope (documented, not implied)

- Versioned doc snapshots per release; localized site content (the site is English,
  like every repo artifact); PDF export; comments/analytics.
- The other Phase 35 legs: Maven Central publication, the Gradle plugin, official
  images.
